# Journal Ranking Project — Context for AI Agents

This document captures all key decisions, architecture, errors, and current status so future AI conversations can pick up where we left off.

---

## 1. Project Overview

**Goal**: Rank academic journals by the fraction of their papers authored by researchers at elite ("tier") institutions — a metric we call "elite revealed preference."

**Intuition**: Top researchers can publish in any journal they choose. The journals they choose to publish in reveal something about journal quality that citation metrics don't capture.

**Two tiers**:
- **Tier 1** (4 institutions): Harvard, Stanford, MIT, Oxford
- **Tier 2** (23 institutions, inclusive of Tier 1): Adds Google, Meta, Microsoft, NIH, US DHHS, Chinese Academy of Sciences, ETH Zurich, UCL, Imperial College London, U of Toronto, UC Berkeley, Princeton, JHU, NUS, Cambridge, Tsinghua, U of Tokyo, Max Planck Society, CNRS

**Data source**: OpenAlex S3 bulk data snapshot (free, no authentication, ~626 GB compressed). We stream directly from S3 — no full download needed.

**Website**: Hosted via GitHub Pages from this repo. Interactive sortable table of journal rankings.

## 3. Data Pipeline

```
OpenAlex S3 snapshot (626 GB compressed, 482M works)
    ↓ extract_works.py (streams, filters, writes parquet parts)
Parquet part files (~13-25 GB for full dataset, 57 fields)
    ↓ analyze.py (filters refs>=10, computes tier rates)
ranking_data.json (~1-2 MB, per-journal rankings)
    ↓ served by GitHub Pages
Website (sortable table with field filters)
```

### Extraction (`extract_works.py`)
- Streams gzipped JSON lines directly from S3 (no disk download)
- Filters: `type == "article"`, `publication_year` 2015-2024, `is_paratext == False`
- Writes to numbered parquet part files (`part_00000.parquet`, etc.)
- Pausable/resumable via JSON checkpoint
- Parallel workers via `ProcessPoolExecutor` (true CPU parallelism)
- Uses `orjson` for 2.2x faster JSON parsing
- CLI: `--workers N`, `--status`, `--merge`, `--reset`

### Analysis (`analyze.py`)
- Loads parquet part files from `snapshot/analysis/`
- Filters: `referenced_works_count >= 10`
- Matches institutions against expanded tier ID sets
- Computes per-journal: paper count, institution coverage, tier rates, field classification
- Outputs `ranking_data.json` at repo root

## 4. Extraction Schema

### Old schema (37 fields, 58 bytes/row compressed)
The data currently in `snapshot/analysis/` uses this schema. Missing: title, author names, institution names/types/lineage, type fields, biblio, topic lists, has_fulltext, ids_pmid, locations_count.

### New schema (57 fields, 165 bytes/row compressed)
Ready to run but full extraction hasn't been done yet. Key additions over old:

| Field | Why added |
|---|---|
| `title` | Spot-checking results |
| `type`, `location_raw_type`, `is_paratext`, `has_fulltext` | Filtering non-research content |
| `ids_pmid` | PubMed presence = signal of real biomedical paper |
| `locations_count` | 0 locations = suspicious |
| `biblio_volume/issue/first_page/last_page` | Missing biblio = signal of non-standard content |
| `first_author_name`, `last_author_name` | Spot-checking |
| `institution_names` | Human-readable institution list |
| `institution_types` | "education", "company", "government" — filter by type |
| `institution_lineage_ids` | **Critical**: flattened parent chain for proper tier matching |
| `topic_ids`, `topic_names`, `topic_scores` | Full topic list, not just primary |
| `source_version` | primary_location.version |

See `openalex-fields.md` for the complete field inventory with sizes and rationale.

## 6. Institution Lineage Problem

**Problem**: OpenAlex does NOT auto-include parent institutions in the `authorships[].institutions[]` array. A paper listing "Max Planck Institute for Solid State Research" will have that sub-institute's ID, NOT the Max Planck Society ID.

**Impact**: Without lineage expansion, we systematically undercount umbrella organizations. Harvard alone has ~14.5% of papers missed without sub-institution expansion.

**Solution (new schema)**: The `institution_lineage_ids` field stores the flattened union of all lineage chains across all authors. Tier matching just checks if the tier institution ID is in this set.

**Solution (old data)**: Since old parquet files lack lineage, `analyze.py` uses hardcoded expanded ID sets — each tier institution maps to a set containing its own ID plus all significant child institution IDs (queried from the OpenAlex API).

**Skipped from expansion**: CAS (100+ sub-institutes), CNRS (hundreds of labs), Max Planck (80+ sub-institutes) — too many children to maintain. These are matched on parent ID only, which undercounts their presence.

## 7. Non-Research Content Filtering

**Problem**: OpenAlex classifies many non-research items (news articles, editorials, letters, conference abstracts) as `type=article`. "Magazine-style" journals like Nature, Science, and The Lancet have 60-80% non-research content classified as articles.

**Primary filter**: `referenced_works_count >= 10`. Research papers cite other work; news articles and editorials typically don't.

**Observed effects**:
- Nature: Tier 1 rate goes from 3.7% (no filter) to 10.3% (refs >= 10)
- refs >= 20 gets Nature to 89% research content
- refs >= 30 gets Nature to 97% research content
- We chose refs >= 10 as a balance between filtering non-research and not excluding legitimate short papers

**Additional signals available** (for future refinement):
- `location_raw_type`: Crossref raw type can distinguish "journal-article" from others
- `biblio_volume/issue/first_page`: Missing biblio suggests non-standard content
- `has_fulltext`: No full text available might indicate a non-article
- `type_crossref` / `location_raw_type`: "journal-article" is the Crossref designation for real articles

## 8. Performance Findings

### JSON parsing
- `orjson` (Rust-based): 2.2x faster than stdlib `json` on our data
- 100K parses/sec vs 45K parses/sec for realistic work objects

### CPU parallelism
- `ThreadPoolExecutor`: Download and gzip parallelize (GIL released), but `json.loads()` holds the GIL. **8 threads is NOT faster than 4** for this workload.
- `ProcessPoolExecutor`: True parallelism — each worker is a separate Python process. **8 workers ≈ 2x faster than 4**.

### Streaming architecture
- `gzip.GzipFile(fileobj=resp.raw)`: Streaming decompression, never loads full file into memory
- Part-file writing: Avoids O(n²) read-concat-rewrite pattern
- Checkpoint batching: Only saves checkpoint when parquet buffer flushes (every 100K rows)

### Throughput
- Network is the bottleneck, not CPU (with ProcessPoolExecutor + orjson)
- At 20 MB/s aggregate: ~8-9 hours for full 626 GB
- At 50 MB/s: ~3-4 hours

## 9. Known Errors & Fixes

| Error | Cause | Fix |
|---|---|---|
| Memory exhaustion on large files | `resp.content` + `gzip.decompress()` loaded entire 900MB file | Streaming decompression with `gzip.GzipFile(fileobj=resp.raw)` |
| O(n²) parquet writes | Read entire file + concat + rewrite on each flush | Write numbered part files, merge at end |
| Too many tiny part files | Flushing after every S3 file | Batch checkpoint saves with the 100K-row flush buffer |
| `TimeoutError` on shutdown | `as_completed(futures, timeout=1)` raised when futures still running | Poll `f.done()` with `time.sleep(0.1)` |
| `ArrowInvalid` error | CSV file in analysis directory confused `ParquetDataset` | Explicitly load only `.parquet` files |
| `ImportError` for `pa` | Used `pa` before import | Moved import to correct location |
| macOS `timeout` command missing | Neither `timeout` nor `gtimeout` available | Used `subprocess.Popen` + `signal.SIGINT` |

## 10. Size & Speed Estimates

### S3 Manifest
- 2,236 files, 482,451,464 total records, 625.9 GB compressed
- File sizes: min ~0 MB, median 258 MB, max 1,149 MB

### Extraction coverage so far
- 643 files processed (smallest first) = 5% of total bytes, 6% of records
- 21,786,585 articles extracted from ~29M records scanned (74.7% match rate)
- **Warning**: The 74.7% match rate is biased — smallest files are unrepresentative
